# 재작업 시 프롬프트 가이드

## 필수 포함 항목

재작업 시 code-writer에게 전달하는 프롬프트에 반드시 포함:

1. **`phase_fail_{n}.md` 파일 경로** — "이 파일을 읽고 이전 실패 패턴을 확인한 뒤 수정해라"
2. **실패 사유 원문** — reviewer/verifier 리포트에서 FAIL 항목 원문 그대로 인용
3. **실패한 파일과 라인 범위** — 구체적인 파일 경로와 라인 번호
4. **이전 시도의 의도** — 이전 code-writer가 왜 그렇게 작성했는지 (알 수 있는 경우)
5. **실패 이유와 수정 방향** — "이전에 X했는데 Y 이유로 FAIL이었으니, Z 방향으로 수정해라"
6. **에스컬레이션 단계에 따른 추가 지시**

## 예시 프롬프트 구조

```
## 수정 요청

### 실패 이력 참조
- phase_fail_{n}.md를 읽고 이전 실패 패턴을 확인해라. 같은 실수를 반복하지 마라.

### 실패 항목 1
- **파일**: src/main/java/com/example/UserService.java:45-60
- **이전 시도**: N+1 쿼리로 User 목록을 조회하도록 작성했음
- **실패 사유** (performance-reviewer 원문): "UserService.findAll()에서 N+1 쿼리 발생. User 엔티티의 roles 연관관계가 LAZY이나 루프 내에서 접근하여 추가 쿼리 N건 발생. CRITICAL."
- **수정 방향**: fetch join 또는 @EntityGraph를 사용하여 한 번의 쿼리로 조회하도록 수정

### 실패 항목 2
...
```

## Reviewer 간 상충 감지

두 reviewer의 권고가 서로 상충하는 경우, 자동 병합하지 않고 사용자에게 인터뷰한다.

**상충 감지 기준:**
- 같은 파일/메서드에 대해 상반된 권고 (예: perf는 "캐싱 추가" vs security는 "민감 데이터 캐싱 금지")
- 한쪽 권고를 따르면 다른 쪽 FAIL 조건을 충족

**상충 발견 시:**
1. 두 권고를 나란히 보여주고 근거와 위험도 요약
2. 사용자에게 우선순위 확인: "성능 우선" / "보안 우선" / "둘 다 해결"
3. 결정에 따라 code-writer에게 전달하거나 handoff Changed에 기록
