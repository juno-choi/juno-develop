# work-review: 업무 진행 검증

현재 프로젝트의 plan, phase, handoff를 종합적으로 검토하여 진행 상황을 보고하고, 계획 대비 변경사항을 식별한다.

## 워크플로우

### 1. 대상 프로젝트 확인

`projects/` 디렉토리에서 대상 프로젝트를 특정한다.
- 사용자가 프로젝트 이름을 지정하면 해당 프로젝트를 사용한다
- 프로젝트가 하나뿐이면 자동 선택한다
- 여러 개인데 지정이 없으면 목록을 보여주고 선택하게 한다

### 2. 전체 문서 읽기

프로젝트의 모든 문서를 읽는다:
- `plan.md` — 전체 계획과 완료 조건
- 모든 `phase_{n}.md` — 각 phase의 Tasks와 Acceptance Criteria
- 모든 `handoff_{n}.md` — 완료/미완료/변경 이력

### 3. 진행 상황 분석

다음 항목들을 검토한다:

**plan 수준:**
- Done When 체크리스트의 달성 현황
- Steps 중 완료된 phase / 진행 중인 phase / 미시작 phase

**phase 수준 (현재 진행 중인 phase 중심):**
- Tasks 체크리스트의 완료 현황
- Acceptance Criteria 달성 여부

**handoff 수준:**
- Blocked 항목이 해소되었는지
- Changed 항목의 영향 범위

### 4. 실제 코드 상태 확인 (선택적)

`context/project_structure.md`에서 대상 코드 프로젝트 경로를 확인한다.
사용자가 코드 레벨 확인을 요청한 경우:
- 해당 프로젝트 디렉토리로 이동하여 git status, 최근 커밋 등을 확인한다
- phase의 Tasks와 실제 코드 변경사항을 대조한다

코드 레벨 확인을 요청하지 않은 경우 문서 기반으로만 리뷰한다.

### 5. 리뷰 보고서 작성

다음 형식으로 사용자에게 보고한다:

```
## 프로젝트 진행 현황: {project_name}

### 전체 진행률
- Plan Steps: {완료}/{전체} phases
- 현재 Phase: phase_{n} — {objective}

### 현재 Phase 상세
- Tasks: {완료}/{전체}
  - ✅ {완료된 task}
  - ⬜ {미완료 task}
- Acceptance Criteria: {달성}/{전체}

### 계획 대비 변경사항
- {변경 내용과 영향}

### Blocked 항목
- {미해결 블로커}

### 권장 다음 액션
- {구체적인 다음 작업 제안}
```

### 6. 문서 업데이트 제안

리뷰 과정에서 발견된 사항이 있으면 업데이트를 제안한다:
- 완료된 Tasks/Acceptance Criteria의 체크 표시
- plan.md의 Done When 체크 업데이트
- 새로 발견된 블로커나 변경사항

사용자 동의 후 문서를 업데이트한다.

## 주의사항

- 리뷰는 현재 상태를 객관적으로 보고하는 것이 목적이다. 임의로 체크리스트를 변경하지 않는다
- 코드 레벨 확인은 사용자 요청 시에만 수행한다 (기본은 문서 기반 리뷰)
- 변경사항이 있으면 단순히 기록하는 것이 아니라, plan 전체에 미치는 영향을 분석한다
